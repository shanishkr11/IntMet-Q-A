1. Encapsulation — protect internal state, expose controlled behaviour ▼

Encapsulation means hiding implementation details inside a class and only exposing what callers genuinely need. Fields are always private — properties are the controlled gateway with validation logic.

Violation vs correct implementation
// BAD — anyone can set Price = -999, no validation
public class Product {
    public decimal Price; // public field — no protection
}

// GOOD — encapsulated with business rule enforcement
public class Product {
    private decimal _price;

    public decimal Price {
        get => _price;
        private set {  // only this class can set it
            if (value < 0)
                throw new ArgumentException("Price cannot be negative.");
            _price = value;
        }
    }

    // The only valid way to change price — business logic in one place
    public void ApplyDiscount(decimal percentage) {
        if (percentage is < 0 or > 100)
            throw new ArgumentOutOfRangeException(nameof(percentage));
        Price = _price - (_price * percentage / 100);
    }
}
Interview answer

"Encapsulation protects invariants. In my e-commerce project, Product.Price was always set through ApplyDiscount() — this guaranteed the 0–100% validation ran consistently, instead of having ad-hoc assignments scattered across controllers and services."

2. Inheritance — base/derived classes, sealed, virtual, override vs new ▼

Inheritance models a true "is-a" relationship. It lets derived classes reuse and extend base class logic. The keywords virtual, override, abstract, and sealed control the behaviour precisely.

E-commerce payment hierarchy
public abstract class Payment {
    public Guid OrderId { get; }
    public decimal Amount { get; }
    protected Payment(Guid orderId, decimal amount) {
        OrderId = orderId; Amount = amount;
    }
    // virtual — derived classes CAN override but don't have to
    public virtual string GetReceipt()
        => $"Order {OrderId} — {Amount:C}";
    // abstract — derived classes MUST implement this
    public abstract Task<bool> ProcessAsync();
}

public class CreditCardPayment : Payment {
    private readonly string _maskedCard;
    public CreditCardPayment(Guid id, decimal amt, string card)
        : base(id, amt) { _maskedCard = "**** " + card[^4..]; }
    public override async Task<bool> ProcessAsync()
        => await CardGateway.ChargeAsync(OrderId, Amount);
    public override string GetReceipt()
        => base.GetReceipt() + $" | Card: {_maskedCard}"; // extends base
}

// sealed — no further subclassing allowed
public sealed class WalletPayment : Payment {
    public WalletPayment(Guid id, decimal amt) : base(id, amt) {}
    public override async Task<bool> ProcessAsync()
        => await WalletService.DeductAsync(OrderId, Amount);
}
override vs new — the key trap

override replaces via vtable — calling through a base reference calls the derived version. new hides the method — calling through a base reference calls the base version. If you write Payment p = new CreditCardPayment() and CreditCardPayment uses new, then p.GetReceipt() calls the base — almost never what you want.

When to use sealed
When subclassing would break your internal assumptions. WalletPayment's logic assumes a fixed deduction flow — any override could corrupt wallet state.
When to use abstract
When a base concept cannot meaningfully exist on its own. You cannot instantiate "Payment" — it must be a specific payment method.
3. Polymorphism — runtime dispatch and compile-time overloading ▼

Polymorphism means the same interface produces different behaviour depending on the actual type at runtime. This eliminates if/else chains and makes your code open to extension.

Runtime polymorphism — CheckoutService knows nothing about payment types
public class CheckoutService {
    public async Task<string> CompleteOrderAsync(Payment payment) {
        // Correct ProcessAsync() called based on ACTUAL type, not declared type
        bool success = await payment.ProcessAsync();
        if (!success) throw new PaymentFailedException();
        return payment.GetReceipt(); // also polymorphic
    }
}
// Same method handles both — no if/else needed
await checkout.CompleteOrderAsync(new CreditCardPayment(...));
await checkout.CompleteOrderAsync(new WalletPayment(...));
Compile-time polymorphism — method overloading
public class OrderService {
    public Order GetOrder(Guid id) { ... }
    public Order GetOrder(string reference) { ... }  // same name, different signature
    public Order GetOrder(Guid id, bool includeItems) { ... }
    // Resolved at compile time — no runtime cost
}
Safe casting with is and as
// is with pattern matching (C# 7+) — preferred
if (payment is CreditCardPayment credit)
    Console.WriteLine(credit._maskedCard); // credit only in scope when cast succeeds

// as — returns null if cast fails (no exception)
var wallet = payment as WalletPayment;
if (wallet != null) Console.WriteLine("Wallet payment");

Prefer is with pattern matching over as. It declares and assigns in one step and the variable is only accessible when the cast succeeds — safer and cleaner.

4. Abstraction — interfaces vs abstract classes, default interface methods (C# 8+) ▼

Abstraction hides how something works and exposes only what it does. In .NET Core this is achieved primarily through interfaces and abstract classes.

Interface — contract for unrelated types
// IExportable can apply to Order and Invoice even though they are unrelated
public interface IExportable {
    byte[] ExportToPdf();
    byte[] ExportToCsv();
}
// ExportService works with ANY exportable — doesn't care what it is
public class ExportService {
    public void Download(IExportable doc, string format) {
        var data = format == "pdf" ? doc.ExportToPdf() : doc.ExportToCsv();
        FileHelper.Save(data, format);
    }
}
Abstract class — shared implementation + is-a relationship
public abstract class PaymentProcessor {
    protected readonly ILogger Logger;
    protected readonly IPaymentAuditRepository AuditRepo;
    protected PaymentProcessor(ILogger logger, IPaymentAuditRepository audit)
        { Logger = logger; AuditRepo = audit; }

    // Template Method — algorithm skeleton is fixed here
    public async Task<PaymentResult> ProcessAsync(PaymentRequest req) {
        Logger.LogInformation("Processing {OrderId}", req.OrderId);
        await ValidateAsync(req);              // shared validation
        var result = await ExecuteAsync(req);  // provider-specific
        await AuditRepo.SaveAsync(req, result);// shared audit
        return result;
    }
    protected virtual async Task ValidateAsync(PaymentRequest req) {
        if (req.Amount <= 0) throw new InvalidPaymentException("Amount invalid.");
    }
    protected abstract Task<PaymentResult> ExecuteAsync(PaymentRequest req);
}
Default interface methods (C# 8+) — backward compatibility
public interface IDiscountStrategy {
    decimal Apply(decimal price);
    // NEW method — existing implementations are NOT broken
    string GetDescription() => $"Strategy: {GetType().Name}";
}
// Old class — compiles fine, uses default GetDescription()
public class SeasonalDiscount : IDiscountStrategy {
    public decimal Apply(decimal price) => price * 0.85m;
    // GetDescription() not written — default version used automatically
}
// IMPORTANT GOTCHA — only accessible through the interface reference
SeasonalDiscount s = new SeasonalDiscount();
s.GetDescription();         // COMPILE ERROR — not visible on concrete type
IDiscountStrategy s2 = s;
s2.GetDescription();        // OK — accessed through interface

5. C# type system — value types, reference types, boxing, nullable, records, generics ▼
Value types vs reference types
// VALUE TYPE (struct) — copied by value, each variable owns its data
public struct Money {
    public decimal Amount { get; }
    public string Currency { get; }
    public Money(decimal amount, string currency) { Amount=amount; Currency=currency; }
    public Money Add(Money other) => new Money(Amount + other.Amount, Currency);
}
// COPY BEHAVIOUR
var price1 = new Money(100, "USD");
var price2 = price1;           // full COPY made
price2 = new Money(200, "USD");
Console.WriteLine(price1);     // USD 100 — unchanged

// REFERENCE TYPE (class) — copy shares the SAME object
var cart1 = new ShoppingCart();
var cart2 = cart1;             // both point to SAME heap object
cart1.AddItem(new CartItem()); 
Console.WriteLine(cart2.Items.Count); // 1 — cart2 sees cart1's change!
Boxing and unboxing — avoid in hot paths
// IMPLICIT BOXING — happens inside non-generic collections and string.Format
ArrayList legacyList = new ArrayList();
legacyList.Add(99.99m);   // decimal → object: BOXING (heap allocation!)
legacyList.Add(42);       // int    → object: BOXING

// FIX — generic collections eliminate boxing entirely
List<decimal> prices = new();
prices.Add(99.99m);       // no boxing

// TRAP — string.Format boxes value types
string bad  = string.Format("Total: {0}", 99.99m); // boxing!
string good = $"Total: {99.99m:C}";                // no boxing (calls ToString directly)
Nullable types — all four operators
var product = await _repo.GetByIdAsync(id); // returns Product?

string category = product?.Category?.Name ?? "Uncategorised"; // ?. + ??
int stock       = product?.StockCount ?? 0;  // null-coalescing default

// ??= null-coalescing assignment (great for caching)
_cache[customerId] ??= new ShoppingCart(); // assign only if null

// ! null-forgiving — use only when YOU are certain it's not null
var customer = cart.GetCustomer()!; // suppress warning, you take responsibility
Records (C# 9+) — value equality + immutability
// Compiler generates: constructor, properties, Equals, GetHashCode, ToString
public record OrderSummary(Guid OrderId, string CustomerName,
                           decimal Total, OrderStatus Status);

// VALUE EQUALITY — unlike classes, same data = equal
var s1 = new OrderSummary(id, "Shanish", 1500m, OrderStatus.Confirmed);
var s2 = new OrderSummary(id, "Shanish", 1500m, OrderStatus.Confirmed);
Console.WriteLine(s1 == s2); // TRUE — records compare by data, not reference

// with expression — non-destructive mutation (original unchanged)
var shipped = s1 with { Status = OrderStatus.Shipped };
Console.WriteLine(s1.Status); // Confirmed — unchanged
Console.WriteLine(shipped.Status); // Shipped — new record
Generics with constraints
// Generic repository — type-safe, no casting, no boxing
public interface IRepository<T> where T : class {
    Task<T?> GetByIdAsync(Guid id);
    Task AddAsync(T entity);
}

// Multiple constraints
public class BaseRepository<T> : IRepository<T>
    where T : class, IEntity, new() { // ref type + interface + parameterless ctor
    protected readonly AppDbContext _db;
    public BaseRepository(AppDbContext db) => _db = db;
    public async Task<T?> GetByIdAsync(Guid id)
        => await _db.Set<T>().FirstOrDefaultAsync(e => e.Id == id);
    public async Task AddAsync(T entity)
        { _db.Set<T>().Add(entity); await _db.SaveChangesAsync(); }
}
// Zero boilerplate — all repos inherit base
public class OrderRepository : BaseRepository<Order> {
    public OrderRepository(AppDbContext db) : base(db) {}
}
When to use records

DTOs, API request/response models, value objects (Money, Address, DateRange). Never for entities with identity (Order, Product, Customer) — use class there.

SRP — Single Responsibility Principle ▼

A class should have one reason to change. If two different business requirements can force you to modify the same class, it has too many responsibilities.

Violation — three jobs in one class
// BAD — OrderService has THREE reasons to change:
// 1. Validation rules change  2. DB/persistence changes  3. Email provider changes
public class OrderService {
    public async Task PlaceOrderAsync(OrderRequest req) {
        if (!req.Items.Any()) throw new ValidationException("Empty cart.");  // Resp 1
        _db.Orders.Add(new Order(req)); await _db.SaveChangesAsync();        // Resp 2
        await _smtp.SendAsync(req.Email, "Confirmed", "...");                // Resp 3
    }
}
Fixed — one class, one responsibility
public class OrderValidator {
    public void Validate(OrderRequest req) {
        if (!req.Items.Any()) throw new ValidationException("Empty cart.");
    }
}
public class OrderRepository : IOrderRepository {
    public async Task<Order> SaveAsync(Order order) { ... }
}
public class OrderNotificationService : IOrderNotificationService {
    public async Task SendConfirmationAsync(Order order, string email) { ... }
}
// Thin orchestrator — delegates, owns no logic itself
public class OrderService : IOrderService {
    public async Task<Order> PlaceOrderAsync(OrderRequest req) {
        _validator.Validate(req);
        var order = new Order(req);
        await _orderRepository.SaveAsync(order);
        await _notificationService.SendConfirmationAsync(order, req.Email);
        return order;
    }
}
Your resume story

"I separated OrderNotificationService from OrderService at Cognine because switching from SMTP to SendGrid should never require touching order logic or its unit tests."

OCP — Open/Closed Principle + Strategy Pattern ▼

Open for extension, closed for modification. Adding new behaviour should not require editing existing tested code.

Violation — if/else grows forever
// BAD — every new payment method requires editing this class
public async Task<bool> ProcessAsync(PaymentRequest req) {
    if (req.Method == "CreditCard") return await ChargeCreditCardAsync(req);
    else if (req.Method == "Wallet") return await DeductWalletAsync(req);
    else if (req.Method == "UPI")    return await ProcessUpiAsync(req); // new edit!
    throw new NotSupportedException();
}
Fixed — Strategy pattern, zero modification for new providers
public interface IPaymentStrategy {
    string Method { get; }
    Task<PaymentResult> ProcessAsync(PaymentRequest req);
}
public class CreditCardPaymentStrategy : IPaymentStrategy {
    public string Method => "CreditCard";
    public async Task<PaymentResult> ProcessAsync(PaymentRequest req)
        => await _gateway.ChargeAsync(req.CardToken, req.Amount);
}
public class UpiPaymentStrategy : IPaymentStrategy { // NEW — zero modification elsewhere
    public string Method => "UPI";
    public async Task<PaymentResult> ProcessAsync(PaymentRequest req)
        => await _upiGateway.InitiateAsync(req.UpiId, req.Amount);
}
// PaymentService NEVER changes
public class PaymentService {
    private readonly Dictionary<string, IPaymentStrategy> _strategies;
    public PaymentService(IEnumerable<IPaymentStrategy> strategies)
        => _strategies = strategies.ToDictionary(s => s.Method);
    public async Task<PaymentResult> ProcessAsync(PaymentRequest req) {
        if (!_strategies.TryGetValue(req.Method, out var s))
            return PaymentResult.Failure("Unsupported method.");
        return await s.ProcessAsync(req);
    }
}
// Adding UPI = one new registration line only
builder.Services.AddScoped<IPaymentStrategy, UpiPaymentStrategy>();
LSP — Liskov Substitution Principle ▼

Derived types must be substitutable for base types without altering correctness. A derived class must honour the base class contract — no stricter preconditions, no unexpected exceptions.

Classic violation — Rectangle → Square
public class Rectangle { public virtual int Width{get;set;} public virtual int Height{get;set;} }
public class Square : Rectangle {
    public override int Width  { set { base.Width=value; base.Height=value; } } // SIDE EFFECT!
    public override int Height { set { base.Width=value; base.Height=value; } }
}
void Resize(Rectangle r) {
    r.Width = 4; r.Height = 5;
    // Rectangle: 4*5 = 20 correct
    // Square: Width setter sets Height=4, then Height=5 sets Width=5 → Area=25 WRONG
}
E-commerce violation — discount adds unexpected precondition
// BAD
public class PremiumDiscount : DiscountStrategy {
    public override decimal Apply(decimal price) {
        if (price < 100) throw new InvalidOperationException("Min price 100 required.");
        // VIOLATION — base only required price > 0. Callers don't expect this exception.
        return price * 0.80m;
    }
}
// FIXED — no extra preconditions, graceful fallback
public class PremiumDiscount : DiscountStrategy {
    public override decimal Apply(decimal price)
        => price >= 100 ? price * 0.80m : price; // no exception — just no discount
}
LSP checklist

No new exceptions the base doesn't throw  ·  No stricter preconditions (accept less)  ·  No weaker postconditions (return less)  ·  If you write if (obj is DerivedType) — that is a LSP smell.

ISP — Interface Segregation Principle ▼

Clients should not be forced to depend on methods they do not use. Prefer many small focused interfaces over one fat interface.

Violation vs fixed
// BAD — fat interface forces all implementors to implement everything
public interface IProductRepository {
    Task<Product?> GetByIdAsync(Guid id);   // read
    Task AddAsync(Product p);               // write
    Task DeleteAsync(Guid id);              // write
    Task UpdateStockAsync(Guid id, int qty); // stock
    Task BulkInsertAsync(IEnumerable<Product> p); // write
}
// ReportingService only reads but drags in write/stock methods

// FIXED — segregated by responsibility
public interface IProductReadRepository  { Task<Product?> GetByIdAsync(Guid id); Task<IEnumerable<Product>> GetAllAsync(); }
public interface IProductWriteRepository { Task AddAsync(Product p); Task UpdateAsync(Product p); Task DeleteAsync(Guid id); }
public interface IProductStockRepository { Task<int> GetStockCountAsync(Guid id); Task UpdateStockAsync(Guid id, int qty); }

// Concrete class implements all — fine for the DB layer
public class ProductRepository : IProductReadRepository, IProductWriteRepository, IProductStockRepository { ... }

// Each service depends ONLY on what it needs
public class ProductReportingService {
    public ProductReportingService(IProductReadRepository readRepo) { ... }
}
public class InventoryService {
    public InventoryService(IProductStockRepository stockRepo) { ... }
}

// Registration — one concrete, multiple interfaces
builder.Services.AddScoped<ProductRepository>();
builder.Services.AddScoped<IProductReadRepository>(sp => sp.GetRequiredService<ProductRepository>());
builder.Services.AddScoped<IProductWriteRepository>(sp => sp.GetRequiredService<ProductRepository>());
DIP — Dependency Inversion Principle + Unit Testing ▼

High-level modules should not depend on low-level modules. Both should depend on abstractions. This is the foundation of ASP.NET Core's DI system.

Violation — tightly coupled, impossible to test
// BAD — OrderService creates its own dependencies (impossible to mock)
public class OrderService {
    private readonly SqlOrderRepository _repo = new SqlOrderRepository();    // concrete
    private readonly SmtpEmailSender _email = new SmtpEmailSender("smtp.."); // concrete
    private readonly StripeGateway _stripe = new StripeGateway("sk_live_.."); // concrete
}
Fixed — depend on abstractions injected from outside
public class OrderService : IOrderService {
    private readonly IOrderRepository _repo;
    private readonly IEmailSender _email;
    private readonly IPaymentGateway _payment;

    public OrderService(IOrderRepository repo, IEmailSender email, IPaymentGateway payment)
        { _repo = repo; _email = email; _payment = payment; }

    public async Task<Result<Order>> PlaceOrderAsync(OrderRequest req) {
        var pay = await _payment.ChargeAsync(req.CardToken, req.Amount);
        if (!pay.IsSuccess) return Result<Order>.Failure(pay.Error!);
        var order = new Order { CustomerId = req.CustomerId, Items = req.Items };
        await _repo.SaveAsync(order);
        await _email.SendAsync(req.Email, $"Order #{order.Id} confirmed", "...");
        return Result<Order>.Success(order);
    }
}

LINQ internals — deferred execution, IQueryable vs IEnumerable, N+1 ▼
Deferred vs immediate execution
// DEFERRED — no DB query happens yet
IQueryable<Order> query = _db.Orders.Where(o => o.CustomerId == id); // deferred
query = query.Where(o => o.Status == OrderStatus.Confirmed);          // still deferred
query = query.OrderByDescending(o => o.CreatedAt);                   // still deferred

// IMMEDIATE — query executes NOW
var orders = await query.ToListAsync();  // single SQL query with all filters
var count  = await query.CountAsync();   // separate optimised COUNT query
IQueryable vs IEnumerable — the most important EF Core distinction
// CORRECT — filter in SQL (IQueryable stays as expression tree)
var highValueOrders = await _db.Orders
    .Where(o => o.Total > 1000)    // WHERE clause in SQL
    .Select(o => new { o.Id, o.Total })
    .ToListAsync();                // single efficient query

// WRONG — materialise THEN filter (entire table loaded into memory)
var allOrders = await _db.Orders.ToListAsync(); // loads ALL rows — could be millions!
var filtered  = allOrders.Where(o => o.Total > 1000); // filter in C# memory
N+1 query problem — most common EF Core performance killer
// BAD — 1 query for orders + 1 query PER order for Customer = N+1
var orders = await _db.Orders.ToListAsync();
foreach (var order in orders)
    Console.WriteLine(order.Customer.Name); // lazy load fires here — 1 query each!

// FIXED — eager loading with Include
var orders = await _db.Orders
    .Include(o => o.Customer)
    .Include(o => o.Items)
    .ToListAsync(); // single JOIN query

// ALTERNATIVE — projection (load only what you need)
var summaries = await _db.Orders
    .Select(o => new { o.Id, CustomerName = o.Customer.Name, o.Total })
    .ToListAsync(); // no navigation property loading at all
Detection tip

Enable EF Core logging: optionsBuilder.LogTo(Console.WriteLine, LogLevel.Information) — N+1 shows up as dozens of identical queries in the log.

async/await — how it works, common mistakes, CancellationToken ▼
How async/await works under the hood
// The compiler transforms this into a STATE MACHINE
public async Task<Order> GetOrderAsync(Guid id) {
    var order = await _repo.GetByIdAsync(id);  // state 1: suspend, free thread
    var items = await _itemRepo.GetAsync(id);  // state 2: suspend, free thread
    return order; // state 3: complete
}
// Thread is RELEASED back to the pool while waiting for I/O
// This is why async scales — 100 concurrent requests don't need 100 blocked threads
Common mistakes — memorise all three
// MISTAKE 1 — async void: exceptions are unobservable, fire-and-forget is dangerous
public async void SendEmail() { ... } // BAD — use async Task instead

// MISTAKE 2 — .Result and .Wait() cause deadlocks in sync contexts
var order = _orderService.GetOrderAsync(id).Result; // DANGEROUS — use await instead

// MISTAKE 3 — sequential awaits when calls are independent
var order    = await _orderRepo.GetAsync(id);       // waits 200ms
var customer = await _customerRepo.GetAsync(cId);   // waits another 200ms = 400ms total

// FIX — parallel execution with Task.WhenAll
var orderTask    = _orderRepo.GetAsync(id);
var customerTask = _customerRepo.GetAsync(cId);
await Task.WhenAll(orderTask, customerTask); // 200ms total — runs in parallel
var order    = await orderTask;
var customer = await customerTask;
CancellationToken — propagate through all layers
// Controller — gets token from ASP.NET Core automatically
[HttpGet("{id}")]
public async Task<IActionResult> GetOrder(Guid id, CancellationToken ct) {
    var order = await _orderService.GetOrderAsync(id, ct);
    return Ok(order);
}
// Service — pass token down
public async Task<Order> GetOrderAsync(Guid id, CancellationToken ct) {
    return await _repo.GetByIdAsync(id, ct); // if client disconnects, query is cancelled
}
// Repository — EF Core honours CancellationToken
public async Task<Order?> GetByIdAsync(Guid id, CancellationToken ct)
    => await _db.Orders.FirstOrDefaultAsync(o => o.Id == id, ct);

ConfigureAwait(false): use in library code to avoid capturing the synchronisation context. In ASP.NET Core application code you don't need it — there is no SynchronizationContext by default.

Service lifetimes — Transient, Scoped, Singleton & captive dependency ▼
Three lifetimes explained
// TRANSIENT — new instance on every injection
// Use for: lightweight, stateless services (validators, mappers)
builder.Services.AddTransient<IOrderValidator, OrderValidator>();

// SCOPED — one instance per HTTP request
// Use for: DbContext, unit-of-work, anything that should be consistent in one request
builder.Services.AddScoped<IOrderRepository, SqlOrderRepository>();
builder.Services.AddDbContext<AppDbContext>(); // DbContext is always Scoped

// SINGLETON — one instance for the entire app lifetime
// Use for: caches, configuration, thread-safe shared services
builder.Services.AddSingleton<IProductCache, InMemoryProductCache>();

Middleware pipeline — Use, Run, Map, filters vs middleware ▼
How Use, Run, and Map differ
var app = builder.Build();

// ORDER IS CRITICAL — top to bottom defines execution order
app.UseExceptionHandler("/error");  // FIRST — catches all exceptions
app.UseHttpsRedirection();
app.UseStaticFiles();
app.UseRouting();
app.UseAuthentication();            // Who are you?
app.UseAuthorization();             // What can you do?
app.MapControllers();

// Use — calls next middleware in chain
app.Use(async (context, next) => {
    Console.WriteLine("Before request");
    await next(context);              // passes to next middleware
    Console.WriteLine("After response");
});

// Run — terminal, never calls next
app.Run(async context => {
    await context.Response.WriteAsync("Final response — pipeline ends here");
});

// Map — branches pipeline for specific path
app.Map("/health", branch => {
    branch.Run(async ctx => await ctx.Response.WriteAsync("Healthy"));
});
Custom middleware — IMiddleware vs inline
// PREFERRED — class-based, testable, DI-friendly
public class RequestLoggingMiddleware : IMiddleware {
    private readonly ILogger<RequestLoggingMiddleware> _logger;
    public RequestLoggingMiddleware(ILogger<RequestLoggingMiddleware> logger)
        => _logger = logger;
    public async Task InvokeAsync(HttpContext context, RequestDelegate next) {
        _logger.LogInformation("{Method} {Path}", context.Request.Method, context.Request.Path);
        await next(context);
        _logger.LogInformation("Responded {StatusCode}", context.Response.StatusCode);
    }
}
// Register
builder.Services.AddTransient<RequestLoggingMiddleware>();
app.UseMiddleware<RequestLoggingMiddleware>();

What is the difference between an abstract class and an interface in C#? When would you choose one over the other?
Medium▼
Answer
An abstract class can have implementation, constructor, and state. An interface defines only a contract (default implementations allowed from C# 8+). Choose abstract class when types share a true "is-a" relationship with common logic — for example a base ClaimProcessor with shared validation and audit logging. Choose interface when unrelated types share a capability — IExportable applied to both Order and Invoice regardless of their inheritance hierarchy.
Resume link: "In the health insurance system, I used an abstract base class for claim processing because all claim types shared validation and logging logic. I exposed them via interfaces so OrderService could use DI and be unit-tested independently."
Explain method hiding (new keyword) vs method overriding (override). What is the runtime behaviour difference?
Hard▼
Answer
override replaces the base method through the vtable — calling through a base reference still dispatches to the derived version. new hides the method — calling through a base reference calls the base version. Example: Payment p = new CreditCardPayment() — if CreditCardPayment uses new GetReceipt(), then p.GetReceipt() calls Payment.GetReceipt(), not CreditCardPayment.GetReceipt(). This is almost never intended. Always use override for polymorphic behaviour.
Interviewers ask this to test whether you understand CLR dispatch, not just syntax. Knowing the vtable mechanism is what earns senior-level credibility.

How does the Dependency Inversion Principle differ from Dependency Injection? Are they the same thing?
Medium▼
Answer
No — they are not the same. DIP is a design principle: high-level modules should depend on abstractions, not on low-level concretions. DI is a technique that helps implement DIP — it supplies the abstractions at runtime via constructor injection. You can use DI while still violating DIP — for example, injecting a concrete SqlOrderRepository directly instead of an IOrderRepository interface. ASP.NET Core's DI container makes injection easy, but choosing to inject abstractions is your responsibility as the developer.
Many developers confuse these. Explaining the distinction clearly — principle vs technique — immediately signals senior-level understanding of design.
What happens when you call .Result on a Task in an ASP.NET Core controller? Why is it dangerous?
Hard▼
Answer
In classic ASP.NET with a SynchronizationContext, calling .Result blocks the current thread while waiting for the Task. If the continuation tries to resume on the same blocked thread, you get a deadlock. In ASP.NET Core there is no SynchronizationContext by default, so deadlocks are less common — but .Result still synchronously blocks a thread pool thread, wasting resources and reducing throughput under load. Always use await. The only acceptable use of .GetAwaiter().GetResult() is in console app entry points or code proven to run outside a sync context.
Follow-up: "What is the difference between Task.Result and Task.GetAwaiter().GetResult()?" — Both block, but GetResult() unwraps exceptions cleanly without the AggregateException wrapper.
What is the difference between IQueryable and IEnumerable in EF Core?
Medium▼
Answer
IQueryable<T> builds an expression tree that EF Core translates into SQL — filtering, ordering, and projection happen in the database. IEnumerable<T> pulls all data into memory first and then filters in C#. The critical mistake is calling ToList() prematurely in a chain, which materialises the query before your Where filter runs. In the 30% performance improvement you made at Cognine, this distinction is directly relevant — a misplaced ToList() before a filter can load millions of rows into memory, negating all stored procedure optimisations.
This is one of the most frequently asked EF Core questions in .NET senior interviews. Also know that AsNoTracking() on IQueryable avoids ChangeTracker overhead for read-only queries — 2–3x faster on large result sets.
Explain the three service lifetimes in ASP.NET Core DI. What is the captive dependency problem?
Medium▼
Answer
Transient: new instance on every injection. Scoped: one instance per HTTP request, shared within the request. Singleton: one instance for the entire application lifetime. The captive dependency problem occurs when a Singleton depends on a Scoped service. Because the Singleton lives forever, it holds the Scoped service beyond its intended lifetime — making it effectively a Singleton too. With EF Core's DbContext this means stale data, concurrency issues, and connection leaks. Fix: inject IServiceScopeFactory into the Singleton and create a new scope explicitly when needed. Enable ValidateScopes = true in development to catch this at startup.
Concrete example: a Singleton background job service injecting a Scoped DbContext — the connection is never released per-request and the context accumulates stale tracked entities over time.
How does the ASP.NET Core middleware pipeline work? Difference between Use, Run, and Map?
Easy▼
Answer
Use adds middleware that calls next — the pipeline continues. Run is terminal — it never calls next and short-circuits the pipeline. Map branches the pipeline based on path. The order of registration determines execution order — exception handling must be first (outermost), authentication before authorization, static files before MVC. Think of the pipeline as nested functions — each middleware wraps the next like Russian dolls.
Draw the nested structure when explaining in interviews: Request → ExceptionHandler → Auth → Routing → Controller → Response. This visual makes the concept instantly clear.
You have multiple notification channels (email, SMS, push). How would you design this using a design pattern?
Medium▼
Answer
Strategy pattern: define INotificationStrategy with Send(NotificationMessage msg). Implement EmailNotificationStrategy, SmsNotificationStrategy, and PushNotificationStrategy separately. A NotificationService receives IEnumerable<INotificationStrategy> via DI and dispatches to all or selects based on user preferences. This satisfies OCP — adding a WhatsApp notification means adding one new class with zero modification to existing code. For testing, each strategy is mocked independently.
Follow-up: "How would you allow a user to choose which channels they receive?" → Store preferences in UserNotificationSettings, filter strategies by user preferences at dispatch time.
What is the Repository pattern? What problem does it solve and what is its biggest criticism?
Easy▼
Answer
Repository abstracts data access behind an interface, decoupling business logic from persistence technology. Benefits: testability (mock the interface), swappability (replace SQL with NoSQL without touching services), and centralised query logic. The main criticism: it can be a leaky abstraction over EF Core because DbContext is already a Unit of Work and Repository. Wrapping it adds boilerplate without value unless you genuinely need to swap data providers or need strict query encapsulation for domain consistency. I use Repository when it adds value — not as a blanket rule.
Saying "I don't always use Repository — only when there is a genuine benefit" signals senior-level pragmatism. Cargo-culting patterns without understanding the trade-off is a junior signal.

SQL Server indexes — clustered, non-clustered, covering, filtered ▼
Index types and their purpose
-- CLUSTERED INDEX — defines physical row order on disk
-- One per table. Primary key is clustered by default.
CREATE CLUSTERED INDEX IX_Orders_Id ON Orders(Id);

-- NON-CLUSTERED INDEX — separate B-tree with pointer to data row
-- Multiple allowed per table. Good for frequently filtered columns.
CREATE NONCLUSTERED INDEX IX_Orders_CustomerId ON Orders(CustomerId);

-- KEY LOOKUP PROBLEM — query selects columns not in the index
-- SQL Server must jump from index to clustered index for each row = expensive
SELECT Id, Total, Status FROM Orders WHERE CustomerId = @cid;
-- Execution plan shows: Index Seek (IX_Orders_CustomerId) → Key Lookup → ...

-- COVERING INDEX — include all needed columns, eliminates key lookup
CREATE NONCLUSTERED INDEX IX_Orders_CustomerId_Covering
ON Orders(CustomerId)
INCLUDE (Total, Status, CreatedAt); -- all selected columns are in the index now
-- Execution plan: Index Seek only — no key lookup

-- FILTERED INDEX — index with WHERE clause, smaller and faster for specific patterns
CREATE NONCLUSTERED INDEX IX_Orders_Active
ON Orders(CustomerId, CreatedAt)
WHERE Status = 'Active'; -- only active orders indexed — much smaller
Reading execution plans — what to look for
OperationMeaningWhat to do
Table ScanNo usable index — reads all rowsAdd an appropriate index
Index SeekTargeted lookup — O(log n)Good — this is what you want
Index ScanReads all rows in indexAdd a more selective filter or covering index
Key LookupExtra trip to clustered indexAdd INCLUDE columns to index
Hash MatchLarge join — building hash table in memoryCheck for missing indexes or statistics
SARGable predicates — critical for index usage
-- NON-SARGABLE — function wraps indexed column → index cannot be used → Table Scan
WHERE YEAR(CreatedAt) = 2024              -- BAD
WHERE CONVERT(VARCHAR, OrderDate) = '...' -- BAD
WHERE UPPER(Email) = 'TEST@TEST.COM'      -- BAD

-- SARGABLE — index can seek directly → fast
WHERE CreatedAt >= '2024-01-01' AND CreatedAt < '2025-01-01'  -- GOOD
WHERE Email = 'test@test.com' COLLATE SQL_Latin1_General_CP1_CI_AS  -- GOOD

-- RULE: never apply functions to the indexed column in WHERE clause
-- Move the function to the value side instead

CTEs, window functions & query optimisation patterns ▼
Window functions — essential for reporting queries
-- ROW_NUMBER — unique sequential, no gaps, no handling of ties
-- RANK       — gaps after ties (1,1,3)
-- DENSE_RANK — no gaps after ties (1,1,2)

-- Find top 3 best-selling products per category
WITH RankedSales AS (
    SELECT p.CategoryId, p.Name, SUM(oi.Quantity) AS TotalSold,
           DENSE_RANK() OVER (PARTITION BY p.CategoryId ORDER BY SUM(oi.Quantity) DESC) AS Rnk
    FROM Products p JOIN OrderItems oi ON p.Id = oi.ProductId
    GROUP BY p.CategoryId, p.Id, p.Name
)
SELECT * FROM RankedSales WHERE Rnk <= 3;

-- Running total of revenue per day
SELECT OrderDate, DailyRevenue,
       SUM(DailyRevenue) OVER (ORDER BY OrderDate ROWS UNBOUNDED PRECEDING) AS RunningTotal
FROM DailyRevenueSummary;

-- LAG/LEAD — access previous/next row values
SELECT OrderDate, Total,
       LAG(Total) OVER (ORDER BY OrderDate) AS PreviousDayTotal,
       Total - LAG(Total) OVER (ORDER BY OrderDate) AS DailyChange
FROM DailyRevenueSummary;
Recursive CTE — category hierarchy
WITH CategoryHierarchy AS (
    -- Anchor: top-level categories (no parent)
    SELECT Id, Name, ParentId, 0 AS Level, CAST(Name AS VARCHAR(500)) AS Path
    FROM Categories WHERE ParentId IS NULL
    UNION ALL
    -- Recursive: join each level to its children
    SELECT c.Id, c.Name, c.ParentId, ch.Level + 1,
           CAST(ch.Path + ' > ' + c.Name AS VARCHAR(500))
    FROM Categories c JOIN CategoryHierarchy ch ON c.ParentId = ch.Id
)
SELECT * FROM CategoryHierarchy ORDER BY Path;
Stored procedures — compilation, parameter sniffing, performance ▼
Parameter sniffing — the hidden performance trap
CREATE PROCEDURE GetOrdersByRegion @RegionId INT AS
    SELECT * FROM Orders WHERE RegionId = @RegionId;
-- First call: GetOrdersByRegion @RegionId = 99 (small region, 100 rows)
-- Optimizer creates plan optimised for 100 rows: Nested Loop join
-- Plan is CACHED for all future calls

-- Later call: GetOrdersByRegion @RegionId = 1 (large region, 1,000,000 rows)
-- CACHED Nested Loop plan runs on 1M rows — catastrophically slow

-- FIX 1: OPTION (RECOMPILE) — new plan every execution (good for infrequent SPs)
SELECT * FROM Orders WHERE RegionId = @RegionId OPTION (RECOMPILE);

-- FIX 2: OPTIMIZE FOR UNKNOWN — ignores sniffed value, uses average statistics
SELECT * FROM Orders WHERE RegionId = @RegionId OPTION (OPTIMIZE FOR (@RegionId UNKNOWN));

-- FIX 3: Local variable technique — optimizer cannot sniff local variables
DECLARE @LocalRegionId INT = @RegionId;
SELECT * FROM Orders WHERE RegionId = @LocalRegionId;

Deadlock — how to prevent and handle
-- CAUSE: Transaction A locks Orders then Products
--        Transaction B locks Products then Orders → circular wait
-- PREVENTION: always access tables in the same order across all transactions

-- C# retry logic for deadlock (error 1205)
public async Task PlaceOrderWithRetryAsync(OrderRequest req, int maxRetries = 3) {
    for (int attempt = 1; attempt <= maxRetries; attempt++) {
        try {
            await _db.ExecuteInTransactionAsync(async () => { ... });
            return; // success
        }
        catch (SqlException ex) when (ex.Number == 1205) { // 1205 = deadlock victim
            if (attempt == maxRetries) throw;
            await Task.Delay(TimeSpan.FromMilliseconds(100 * attempt)); // exponential backoff
        }
    }
}

Refresh token rotation — secure token lifecycle
// ACCESS TOKEN — short-lived (15 min), stateless JWT
// REFRESH TOKEN — long-lived (30 days), stored in DB + HttpOnly cookie

// Flow: access token expires → client sends refresh token → server issues new pair
public async Task<AuthTokens> RefreshAsync(string refreshToken) {
    var stored = await _tokenRepo.GetAsync(refreshToken);
    if (stored == null || stored.ExpiresAt < DateTime.UtcNow)
        throw new UnauthorizedException("Refresh token invalid or expired.");

    // ROTATION — old refresh token is invalidated, new one issued
    // If old token is reused (theft detected), revoke the entire token family
    if (stored.IsUsed) {
        await _tokenRepo.RevokeTokenFamilyAsync(stored.FamilyId); // detect theft
        throw new SecurityException("Token reuse detected — all sessions revoked.");
    }
    stored.IsUsed = true;
    await _tokenRepo.UpdateAsync(stored);

    var newAccessToken  = _jwtService.GenerateAccessToken(stored.UserId);
    var newRefreshToken = _jwtService.GenerateRefreshToken(stored.FamilyId); // same family
    await _tokenRepo.SaveAsync(newRefreshToken);
    return new AuthTokens(newAccessToken, newRefreshToken);
}
Token revocation challenge

JWTs cannot be invalidated before expiry because the server has no record of issued tokens. Solutions: (1) Short expiry + refresh rotation (most common). (2) Redis blacklist for JTI claim. (3) Token versioning — store a tokenVersion integer on the user record, include in JWT claim, increment on logout/password change. Any token with an old version is rejected.

What is the difference between clustered and non-clustered index? When would you use a covering index?
Medium▼
Answer
A clustered index defines the physical storage order of rows — one per table, usually the primary key. Non-clustered indexes are separate B-tree structures with pointers back to data rows. When a query selects columns not present in the non-clustered index, SQL Server performs a key lookup back to the clustered index for each row — expensive at scale. A covering index solves this by adding needed columns with INCLUDE, satisfying the entire query from the index without touching the base table. This directly applies to your 30% improvement story — key lookups were the bottleneck.
Always connect index questions to your resume: "I identified key lookup operations in the execution plan and added INCLUDE columns. This alone dropped the query cost by over 50% in the RAB application."
What is a SARGable predicate? Give an example of a non-SARGable query and how to fix it.
Hard▼
Answer
SARGable (Search ARGument ABLE) means the query optimizer can use an index seek for the predicate. Applying a function to an indexed column makes it non-SARGable and forces a table scan. Classic violation: WHERE YEAR(CreatedAt) = 2024 — rewrite as WHERE CreatedAt >= '2024-01-01' AND CreatedAt < '2025-01-01'. Other traps: UPPER(Email) = '...', Price * 1.1 > 100. Rule: never apply functions to the indexed column side — move transformations to the value side.
Knowing the term "SARGable" immediately signals strong SQL proficiency. Most developers know to avoid functions on columns but don't know the technical term for why.
How would you find the second highest order total per customer in SQL Server?
Medium▼
Answer
Use DENSE_RANK() — it handles ties correctly unlike ROW_NUMBER():

WITH RankedOrders AS (SELECT CustomerId, TotalAmount, DENSE_RANK() OVER (PARTITION BY CustomerId ORDER BY TotalAmount DESC) AS Rnk FROM Orders) SELECT CustomerId, TotalAmount FROM RankedOrders WHERE Rnk = 2

Using ROW_NUMBER() instead would skip the second result if two orders have the same highest amount — a common mistake. RANK() would also work but produces gaps in numbering after ties.
Interviewers follow up with: "What is the difference between ROW_NUMBER, RANK, and DENSE_RANK?" — ROW_NUMBER: always unique sequential. RANK: gaps after ties (1,1,3). DENSE_RANK: no gaps (1,1,2).
What is parameter sniffing in SQL Server? How does it cause problems and what are the fixes?
Hard▼
Answer
When a stored procedure compiles for the first time, SQL Server builds a plan based on the parameter values at that moment. If data is skewed, this cached plan may be perfect for small datasets but catastrophic for large ones. Example: first call for a small region (100 rows) → Nested Loop plan cached. Next call for a large region (1M rows) → same Nested Loop plan used → devastating performance. Three fixes: (1) OPTION(RECOMPILE) — new plan every call, good for infrequent SPs with high variance. (2) OPTIMIZE FOR UNKNOWN — uses average statistics, ignores sniffed values. (3) Local variable technique — assign parameter to local variable; optimizer cannot sniff locals.
Knowing all three fixes and their trade-offs is the senior-level answer. RECOMPILE adds compilation cost — use it judiciously. OPTIMIZE FOR UNKNOWN may produce mediocre plans for everyone. Local variables are zero-cost but may miss optimal plans for well-distributed data.
Explain ACID properties with an example from an order processing system.
Easy▼
Answer
Atomicity: deducting stock and inserting the order must both succeed or both fail — no partial state. Consistency: the database always moves between valid states — stock cannot go negative (enforced by CHECK constraint). Isolation: two customers buying the last item simultaneously must not both succeed — SERIALIZABLE isolation ensures their transactions appear sequential. Durability: once an order is committed, it survives a server crash — guaranteed by the SQL Server transaction log which is flushed to disk before COMMIT returns. In code: wrap stock deduction and order insert inside a single BEGIN TRANSACTION...COMMIT block.
Always use a concrete order processing scenario rather than abstract definitions. It shows you understand why ACID matters in your actual work, not just what the letters stand for.
What is idempotency in REST APIs? Why does it matter for order placement?
Medium▼
Answer
An idempotent operation produces the same result regardless of how many times it is called. GET, PUT, and DELETE are idempotent. POST is not by default. In order placement, if a client POSTs an order and the network times out, the client retries — but the first request may have already succeeded, creating a duplicate order. Solution: the client sends a unique Idempotency-Key header (GUID). The server stores processed keys in Redis with a 24-hour TTL. On retry, the server returns the original response without reprocessing. This pattern is used by Stripe, PayPal, and virtually all payment APIs.
Follow-up: "Where would you store idempotency keys?" → Redis is preferred over SQL for this — sub-millisecond lookup with automatic TTL expiry, no table bloat.
What is the difference between 401 Unauthorised and 403 Forbidden?
Easy▼
Answer
401 means the request lacks valid authentication credentials — the server does not know who you are. The client should authenticate and retry. 403 means authentication succeeded but the authenticated user lacks permission — the server knows who you are and refuses access. In e-commerce: accessing /api/admin/products without a JWT → 401. Accessing it with a valid customer JWT (not admin role) → 403.
Mnemonic: 401 = "Who are you?" · 403 = "I know who you are, but no." This distinction trips up many developers. Knowing it precisely signals attention to API design standards.
What is the difference between offset pagination and cursor pagination? When would you use each?
Medium▼
Answer
Offset pagination uses SKIP(n)/TAKE(m). Simple to implement but degrades at large offsets — SQL must count through all skipped rows even if none are returned. Also produces duplicates or skipped rows if data is inserted between pages. Cursor pagination uses a stable pointer (last OrderId seen): WHERE Id > @lastId ORDER BY Id LIMIT @pageSize. O(log n) regardless of depth, consistent with real-time data. Use cursor for product listings, order history, and any data that changes. Use offset for admin panels with static datasets where pagination depth is bounded.
Senior answer: mention that EF Core's Skip(n) translates to OFFSET n ROWS in SQL — the database must count through all n skipped rows. At page 500 with 20 items per page, that is counting 10,000 rows before returning 20.
Is JWT payload data secure? What should and should not be stored in a JWT?
Medium▼
Answer
No — the JWT payload is Base64Url encoded, not encrypted. Anyone with the token can decode and read the payload without the signing key. The signature only proves the token was not tampered with — it does not hide data. Store: UserId, Role, Email, CustomerId, Permissions, standard claims (sub, iat, exp, iss, aud). Never store: passwords, card numbers, sensitive PII, secrets of any kind. If you need to transmit sensitive encrypted data, use JWE (JSON Web Encryption) — but this is uncommon in standard web APIs.
Resume connection: "In my e-commerce project I stored UserId, Role, and CustomerId in JWT claims. Admin routes checked the Role claim via a policy. Payment card data never touched the token — it went through the payment gateway directly."
How do you invalidate a JWT before it expires? What are the trade-offs of each approach?
Hard▼
Answer
JWTs are stateless — the server has no record of issued tokens, so direct invalidation is not possible. Three approaches: (1) Short expiry + refresh rotation — 15-minute access tokens limit damage window without server-side state. Simplest. (2) Redis blacklist — store revoked JTI claims, check on every request. Immediate revocation but adds latency and introduces server state. (3) Token versioning — store a tokenVersion integer per user in the database. Include in JWT claim. On logout or password change, increment the version. Any token with an older version is rejected. This is the most practical solution — minimal state (one integer per user), immediate revocation, no Redis dependency.
This question tests understanding of the statelessness trade-off. The expected answer at senior level demonstrates knowing the constraint and the three industry patterns — not just "use short expiry."
What is the N+1 query problem in EF Core? How do you detect and fix it?
Medium▼
Answer
N+1 occurs when you load a collection (1 query) and then access a navigation property inside a loop, causing one additional query per item (N queries total). Loading 100 orders then accessing order.Customer.Name in a loop = 101 queries. Fix with eager loading: _db.Orders.Include(o => o.Customer).ToListAsync(). Alternative: projection with Select to load only needed columns without navigation properties — fastest approach for read-only queries. Detection: enable EF Core logging in development to see all SQL queries. Tools like MiniProfiler also surface N+1 clearly.
Advanced follow-up: "What is AsSplitQuery()?" → For collections with Include, EF Core 5+ can split the query into separate SQL statements instead of one large JOIN, avoiding Cartesian explosion when including multiple collections.
What is the difference between tracked and no-tracking queries in EF Core?
Easy▼
Answer
By default EF Core tracks every loaded entity in the DbContext ChangeTracker — it snapshots entity state to detect modifications for SaveChanges. This costs memory and CPU. For read-only scenarios (GET endpoints, reports, dashboards) use AsNoTracking() — it bypasses the ChangeTracker entirely, making queries 2–3x faster on large result sets. Rule: use tracking only when you intend to modify and save the entity. Use AsNoTracking() for all read-only queries. AsNoTrackingWithIdentityResolution() handles related entities without full tracking overhead when you need consistent object references in a query result.
Quick win: audit your GET endpoints and add AsNoTracking() everywhere you don't call SaveChanges on the result. This is a low-effort, high-impact performance improvement in any EF Core codebase.
